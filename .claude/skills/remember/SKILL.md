# Remember — Knowledge Capture

从对话中提取知识，保存到 `D:/AIprogram/myaitool/knowledge/`。

## Trigger

用户调用 `/remember`，或说"保存这个经验"、"记录下来"等。

## Mode Detection

检查 `$ARGUMENTS` 是否为空：

- **有参数** → 显式模式：解析分类和内容，直接保存
- **无参数** → 自动提取模式：分析对话，建议要保存的洞察

## 显式模式

解析 `$ARGUMENTS` 为：`<category> <content>`

有效分类以 hub 索引为准：先读取 `D:/AIprogram/myaitool/knowledge/INDEX.md`，从索引中的相对路径选择目标文件。

关键词选择规则：

- 如果第一个词能匹配索引里的分类名、文件名或说明（如 `git`、`godot`、`vite`、`websocket`、`windows`），使用对应索引路径。
- 如果匹配到多个候选，优先选择最具体的文件（例如 `git` → `general/git-workflow.md`，`vite` → `web/vite-proxy.md`）。
- 如果没有明确匹配，从内容推断最佳分类并确认。

### 保存步骤

1. 读取目标文件（不存在则创建）
2. 生成简洁标题（60 字以内）
3. 在文件末尾追加新条目
4. 根据内容类型选择格式：
   - **反模式**（错误做法、失败尝试）：
     ```
     ## {date}: {title}
     - **标签**: #tag1 #tag2
     - **陷阱**: 错误的做法
     - **为什么容易犯**: 为什么看起来是对的
     - **后果**: 具体的失败模式
     - **正确做法**: 正确的模式
     - **如何提前发现**: 危险信号
     ```
   - **问题解决**（bug/修复）：
     ```
     ## {date}: {title}
     - **标签**: #tag1 #tag2
     - **症状**: 出了什么问题
     - **排查**: 尝试了什么，走了哪些弯路
     - **根因**: 实际原因
     - **修复**: 具体改动
     - **如何应用**: 何时需要注意
     ```
   - **通用洞察**：
     ```
     ## {date}: {title}
     - **标签**: #tag1 #tag2
     - **经验**: 具体内容
     - **How to apply**: 如何应用
     ```
5. 更新 `knowledge/INDEX.md`（如有新文件）
6. 确认保存结果

## 自动提取模式

分析当前对话，识别有价值的项目知识。

### 识别信号

1. **问题解决对**（最高优先级）— 需要调查的 bug、错误或意外行为
2. **反模式**（最高优先级）— 看似正确但实际错误的做法
3. **决策** — "选了 X 而不是 Y，因为 Z"
4. **发现的模式** — "原来...", "正确的方式是..."
5. **陷阱和坑** — "这个不行", "注意", "错误是由...引起的"
6. **配置洞察** — 服务器设置、网络配置、环境变量
7. **架构洞察** — 系统如何交互、数据流、服务依赖
8. **调试发现** — 找到的根因、解码的错误信息、应用的变通方案
9. **性能发现** — 什么慢、什么有帮助
10. **自动化知识** — 脚本、批处理、工作流优化

### 跳过内容

- 没有更广泛洞察的常规代码更改
- CLAUDE.md 或现有知识文件中已捕获的信息
- 临时调试状态（变量值、堆栈跟踪）除非揭示了模式
- 属于 auto-memory 的用户偏好
- **纯项目特定内容**（见下方过滤规则）

### Hub 过滤规则（重要）

Hub 知识库只存**通用/跨项目**经验。以下内容**不入 hub**，应写入项目本地 `.shared/knowledge/`：

| 判断依据 | 入 hub | 不入 hub（写 .shared/knowledge/） |
|----------|--------|----------------------------------|
| 涉及的文件/类/函数 | 通用模式、框架特性 | 项目特定的文件名、类名、业务逻辑 |
| Bug 描述 | 框架/工具的通用 bug 模式 | 项目代码里的具体 bug |
| 配置 | 通用配置模式（Vite、Webpack 等） | 项目的具体配置值 |
| 架构 | 通用设计模式 | 项目的具体架构决策 |
| 调试经验 | 通用调试技巧、工具用法 | 项目特定的排查过程 |

**示例**：
- ✅ 入 hub："Vite proxy 在 iframe 中不转发 WS"（通用模式）
- ❌ 不入 hub → 写 .shared/knowledge/："control-panel 的 login API 返回 500 因为 env 变量没配"（项目特定）
- ✅ 入 hub："Windows exec 双重壳层导致 DLL 加载失败"（通用 Windows 问题）
- ❌ 不入 hub → 写 .shared/knowledge/："slg-card-test 的战斗系统在第 3 回合崩溃"（项目特定）

如果判断不确定，用 `AskUserQuestion` 问用户。

### 处理流程

1. 扫描对话中的信号
2. 对每个候选洞察确定：类型、分类、标题、内容
3. 读取现有知识文件检查重复
4. 展示给用户确认：
   ```
   发现 N 个值得保存的洞察：

   1. [分类] ⚠️ 标题 — 一句话摘要
      陷阱: ...
      正确做法: ...
   2. [分类] 标题 — 一句话摘要
      问题: ...
      解决: ...

   全部保存？或指定编号（如 "1,3"）：
   ```
5. 确认后保存到对应文件
6. 确认保存结果

## 文件操作

- 文件不存在时创建，标题为 `# {Category}`
- 读取文件检查重复
- 新条目追加到文件末尾
- 使用 `Edit` 工具追加（保留现有内容）
- 使用绝对路径

## 知识格式

```markdown
## YYYY-MM-DD: 标题

- **标签**: #tag1 #tag2
- **上下文**: 什么场景
- **经验**: 具体内容
- **How to apply**: 如何应用
```
